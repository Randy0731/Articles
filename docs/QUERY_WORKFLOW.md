# 查詢與交叉比對流程

本流程用於使用者問既有資料，例如 ticker、主題、KOL 觀點、框架、時間區間、觀點變化。

## 1. 查詢前先確認資料範圍

先讀：

- `indexes/source_index.md`
- 相關 KOL 資料夾
- 相關索引檔
- 必要時讀 `docs/TAG_TAXONOMY.md`，用標準 tag 查同義主題

若查大叔，預設查 `KOL/大叔美股筆記/notes/` 與 `KOL/大叔美股筆記/articles/`。只有使用者明確指定聊天室資料時，才查聊天室逐字稿或對話分析。

若查詢日期或資料不存在，直接說沒有資料。

合併筆記內文可能是 append-only，不代表時間順序。查詢時以文章 metadata、頂部文章索引與全域索引的日期欄位排序，不用內文所在位置判斷新舊。

## 2. 查詢類型

| 類型 | 範例 | 查法 |
|---|---|---|
| 個股 | `MU 最近怎麼樣` | ticker index -> 相關 KOL 筆記 -> 日期倒序 |
| KOL 觀點 | `大叔對 memory 的看法` | 該 KOL notes / articles |
| 主題 / 板塊 | `先進封裝脈絡` | theme index + framework index |
| 框架 | `宋分哪幾篇講 CCC` | framework index -> 原文 |
| 時間區間 | `過去三天重點` | source index + 每日結論 |
| 觀點變化 | `有沒有改口` | 同 KOL 依日期排序 |
| 催化劑 | `ONDS 接下來看什麼` | catalyst index + ticker index |

## 3. 同 KOL 交叉比對

預設只比對同一 KOL。

輸出時依序呈現：

1. 最新結論
2. 日期倒序觀點表
3. 觀點是否改變
4. 信心強度是否改變
5. 時間框架是否不同
6. 價位性質是否不同
7. 尚未解決的矛盾或風險

不要把不同 KOL 的框架混成單一觀點。

## 4. 跨 KOL 比較

只有使用者明確要求時才做。

格式：

| KOL | 觀點 | 訊號強度 | 時間框架 | 主要證據 | 風險 |
|---|---|---|---|---|---|

最後可以提供「共同點 / 分歧點」，但不能寫成「市場共識」。

跨 KOL 比較預設只在對話中回答，不寫入新檔。只有使用者明確說「整理成比較筆記」「幫我入庫」時，才寫入 `Research/comparisons/` 或指定位置。

跨 KOL 比較時：

- 各 KOL 觀點分欄呈現
- 相似框架只能標「相近」，不能合併成同一觀點
- 不把 A KOL 的框架套成 B KOL 的原始觀點

## 5. 引文與證據

優先使用已驗證原文。

聊天室格式：

```text
KOL名稱（2026-04-22, p.145）：「...」
```

文章格式：

```text
KOL名稱（2026-04-16, 《文章標題》）：「...」
```

若只有整理者摘要，不能加引號，只能寫「整理筆記顯示」。

## 5.5 外部資料與現況補充

若查詢需要最新股價、最新財報日、最新分析師目標價或公司狀態，必須另外標明查詢日期與來源。

回答時分成：

- **原文 / KOL 觀點**：只來自筆記與原文
- **整理日補充**：來自外部查詢或最新資料

不可把整理日補充寫成 KOL 當時的原始觀點。

## 5.6 催化劑查詢維護

查詢催化劑、個股儀表板或「接下來看什麼」時：

- 先檢查 `indexes/catalyst_index.md` 中事件日期是否已過。
- 已過期但專案沒有結果資料時，回答「待確認」，不要推測結果。
- 若使用外部資料補結果，必須標明查詢日期與來源，並分開寫在「整理日補充」。
- 若使用者要求入庫或本次同時有新資料入庫，再同步更新 catalyst index 與個股儀表板狀態。

## 6. 回答格式

一般查詢：

```text
結論：

| 日期 | 來源 | 觀點 | 訊號強度 | 風險 / 催化劑 |
|---|---|---|---|---|

重點拆解：

風險：

一句話結論：
```

個股查詢：

```text
結論：

| KOL / 日期 | 立場 | 訊號強度 | 價位 | 催化劑 | 風險 |
|---|---|---|---|---|---|

觀點變化：

可追蹤事件：

一句話結論：
```

## 7. 沒有資料時

直接說：

```text
目前專案內沒有 [KOL / 日期 / ticker / 主題] 的資料，所以不能判斷。
```

可以補充有哪些相近資料，但不可用相近資料冒充答案。
