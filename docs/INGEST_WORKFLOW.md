# 新文章入庫流程

本流程用於使用者上傳文章、PDF、截圖、連結、短 Note、聊天紀錄或研究報告時。

## 0. 啟動前檢查

1. 讀 [PROJECT_RULES.md](PROJECT_RULES.md)
2. 讀 [kol_config.yaml](kol_config.yaml)
3. 讀 [docs/KOL_PROFILES.md](docs/KOL_PROFILES.md)
4. 讀 [docs/STOCK_PROFILES.md](docs/STOCK_PROFILES.md)
5. 讀 [docs/FORMAT_SPEC.md](docs/FORMAT_SPEC.md)
6. 讀 [docs/TAG_TAXONOMY.md](docs/TAG_TAXONOMY.md)
7. 查看相關 KOL / Stock / Research 資料夾與索引是否已有重複資料

## 0.5 通用偵測

所有新資料都先偵測，再整理。偵測是入庫流程的一部分，不是預設停下來等待確認。

偵測輸出至少包含：

- `source_id`
- KOL / 來源
- 檔案格式與處理方式
- 文章類型
- 標題與日期
- 內容完整性：是否疑似截斷、缺頁、裁切、前後文不足或核心內容不清楚
- raw 複本路徑 / 原始路徑 / URL
- OCR 狀態
- 相關 ticker / 主題
- ticker 狀態：已追蹤 / 僅索引 / 候選追蹤
- 重要性層級（一般 KOL 筆記、個股 L1-L4 交會、框架、催化劑）
- 預計寫入路徑
- 是否疑似重複

若涉及已追蹤個股，依 [docs/STOCK_PROFILES.md](docs/STOCK_PROFILES.md) 判斷 L0-L4 交會等級。

若出現新 ticker，依 [docs/STOCK_PROFILES.md](docs/STOCK_PROFILES.md) 的「新 ticker 偵測與升級流程」處理；預設只更新 `ticker_index.md`，不要自動建立 `Stocks/<Ticker>/`。

偵測完成後，若使用者沒有明確禁止整理，直接進入後續入庫流程。

### 0.6 內容完整性阻斷

在進入正式入庫前，先判斷內容是否足以可靠整理。只要偵測到以下任一情況，立即終止入庫並通知使用者：

- 文章或貼文明顯被截斷，缺少開頭、結尾或核心段落
- PDF 缺頁、頁序不明，或 OCR 只抓到片段
- 截圖被裁切、解析度過低、重疊遮擋，導致核心內容不清楚
- 聊天室紀錄缺日期、頁碼或上下文，導致無法判斷說話者與語境
- 無法可靠判斷主旨、KOL 立場、ticker、價位、催化劑、引文來源或原文脈絡

終止入庫時：

1. 不建立整理筆記。
2. 不更新 KOL / Stock / Research 資料夾。
3. 不更新 `indexes/` 或 `logs/update_log.md`。
4. 回覆使用者：`內容疑似截斷 / 不清楚，已停止紀錄`。
5. 簡短說明需要使用者補充的內容，例如完整截圖、原始連結、完整 PDF、缺漏頁面或更清晰版本。

只有在少量 OCR 缺字或非核心表格瑕疵不影響主旨、日期、立場、ticker 與引文可追溯性時，才可繼續入庫，並在筆記中標明資料限制。

只在以下情況先問使用者：

- KOL / 來源判斷不明
- 多篇文件重複或順序不清
- 可能覆蓋既有筆記
- 原文無法讀取，需要補檔或改用其他讀取方式
- 原文疑似被截斷或內容不清楚，需要使用者補齊；此時不繼續入庫
- 使用者正在討論規則、概念或問題，並非要求入庫

## 1. 判斷來源與 KOL

判斷依據優先順序：

1. 使用者明確指定
2. 檔名
3. 文章標題 / 作者 / 來源 URL
4. 內容語氣與專用框架
5. 無法判斷時歸入 `其他KOL`，並在筆記中標明「KOL 未確認」

## 2. 判斷內容型態

| 型態 | 處理方式 |
|---|---|
| 短 Note | 合併至該 KOL 主筆記 |
| 長文 | 獨立成文 |
| PDF 文章 | 先抽文字；必要時 OCR |
| 截圖 | OCR 後整理 |
| 聊天室 | 依日期建立對話分析，保留頁碼 |
| 財報分析 | 建立財報重點、guidance、風險、催化劑 |
| 純框架文 | 更新 framework index |
| 個股快評 | 更新 ticker index |

短文 / 長文預設分界：

- 整理後 <= 150 行：短文
- 整理後 > 150 行：長文

若 KOL 專屬規則不同，以 KOL 專屬規則為準。

已追蹤個股的長短文分界：

- 一般 L2 / L3：進季度檔或 notes
- > 200 行：進 `longform/`
- L4：無論長短都進 `milestones/`

## 3. 原始檔處理

### 3.1 建立 source_id

所有新入庫資料先建立 `source_id`：

```text
KOL_OR_SOURCE-YYYYMMDD-title-slug-short_hash
```

優先用原文發文日期；若未知，使用整理日期並在筆記中標明發文時間未確認。`short_hash` 由 URL、原始檔名或檔案內容摘要產生，用於同日同名去重。

若使用者提供本機檔案路徑：

1. 記錄原始路徑到筆記 metadata
2. PDF、截圖、Markdown、文字檔等實體檔案，複製一份到對應 `raw/`
3. raw 檔使用標準命名：

   ```text
   YYYYMMDD_KOL_OR_SOURCE_title_slug_source_id.ext
   ```

4. 不修改原始檔

若是網頁連結：

1. 記錄 URL、抓取日期與來源平台
2. 不需要複製到 `raw/`

若原始檔很大或不適合複製：

1. 保留原始路徑
2. 在筆記 metadata 標明「未複製 raw，僅記錄原始路徑」

若是 PDF：

1. 用 `file` 判斷格式
2. 真 PDF：先文字抽取
3. 文字不足：改用 OCR
4. ZIP 偽裝 PDF：解壓圖片後依序 OCR
5. 聊天室 PDF：保留頁碼與訊息位置
6. 記錄 OCR 狀態：不適用 / 未 OCR / 完整 / 部分 / 不可靠

OCR 狀態判定：

| 狀態 | 使用情境 |
|---|---|
| 不適用 | 可讀文字、Markdown、網頁文字或使用者貼文 |
| 未 OCR | 只保存 raw，尚未辨識 |
| 完整 | OCR 足以支撐完整整理 |
| 部分 | 有少量缺字、表格缺字、圖片局部低解析或少量辨識問題，但不影響核心判斷 |
| 不可靠 | OCR 文字不足以支撐明確結論，或核心內容疑似截斷 / 不清楚；必須終止入庫並通知使用者 |

若內容完整性已觸發阻斷，不得用「低信心摘要」替代入庫筆記。

## 4. 去重檢查

入庫前檢查：

- `source_id` 是否已存在於 `indexes/source_index.md`
- 標題是否已存在
- URL 是否已存在
- 原始檔名是否已存在
- 發文時間與主題是否高度重疊
- ticker / 主題 / 關鍵金句是否疑似重複

若疑似重複：

- 不直接覆蓋
- 回報「疑似已入庫」
- 若只是補充版，建立「修正版」或更新既有文章並記錄到 update log

補充版 / 修正版處理：

- 同一篇文章的 OCR 改善、補圖、補頁、錯字修正：更新原筆記，不新開一篇
- 標題相同但來源日期、作者或內容明顯不同：保留為新篇
- 更新原筆記時，在文末 `修訂紀錄` 或 update log 記錄修正原因
- `source_index.md` 保留同一 `source_id`，在狀態欄標記「修正版」或「補充版」

## 5. 筆記生成

依 [docs/FORMAT_SPEC.md](docs/FORMAT_SPEC.md) 選擇格式；需要範例時參考 [docs/TEMPLATES.md](docs/TEMPLATES.md)。

每篇至少包含：

- source_id
- 基本資訊
- raw 路徑 / URL
- OCR 狀態
- 主旨
- 個股觀點（若有）
- 核心框架 / 心法
- 關鍵證據 / 引文（若有）
- 風險與反例
- 延伸追蹤（若有）

不要硬湊空區塊。沒有就省略或明確寫「本篇無」。

若本篇是修正版或補充版，文末增加：

```md
### 修訂紀錄

| 日期 | 類型 | 說明 |
|---|---|---|
| YYYY-MM-DD | 修正版 / 補充版 / OCR 改善 | ... |
```

### 5.5 合併筆記寫入順序

若本篇要寫入 KOL 主筆記或個股季度檔等合併筆記：

1. 不重排既有內文。
2. 將新整理文章追加到檔尾。
3. 更新檔案頂部的 `文章索引（按發文時間倒序）`。
4. 文章索引用 `發文時間` 倒序；若發文時間不明，用 `整理日期` 排序並標「未確認」。
5. 補舊文仍追加檔尾，文章索引備註寫「補入舊文」，update log 同步記錄。
6. 查詢時以文章 metadata、source index 與文章索引排序，不以內文物理位置判斷新舊。

## 6. 索引更新

每次入庫後視內容更新：

| 索引 | 何時更新 |
|---|---|
| `indexes/ticker_index.md` | 有 ticker 或公司 |
| `indexes/theme_index.md` | 有產業、主題、宏觀框架 |
| `indexes/framework_index.md` | 有可複用方法論 |
| `indexes/catalyst_index.md` | 有明確日期或事件 |
| `indexes/source_index.md` | 新來源、新檔案、新 URL |
| `indexes/watchlist_index.md` | 新 ticker 值得觀察但尚未升級為個股專案 |
| `logs/update_log.md` | 每次入庫都要更新 |

`source_index.md` 必須寫入 `source_id`、raw 路徑、OCR 狀態與狀態欄，作為去重的第一查詢點。

### 6.1 催化劑狀態維護

每次入庫後若更新 `catalyst_index.md` 或個股儀表板：

1. 檢查既有催化劑日期是否已過。
2. 已過期但結果未知，標記「待確認」。
3. 有結果時標記「已兌現 / 未兌現 / 延期」，並補結果來源。
4. 儀表板只保留近 30 天待追蹤與近 90 天已兌現；完整歷史保留在 `catalyst_index.md`。

## 6.5 已追蹤個股同步

若新文章對已追蹤個股達 L2 以上：

1. 原文主整理仍放在來源所屬位置，例如 `KOL/KP_FOMOSoc/weekly/`。
2. 對每個 ticker 獨立判斷 L0-L4，不整篇文章套同一等級。
3. 依 [docs/STOCK_PROFILES.md](docs/STOCK_PROFILES.md) 的季度歸檔規則判斷季度：優先用發文時間，其次事件日期，最後整理日期。
4. 到 `Stocks/<Ticker>/` 寫入摘要型個股筆記、季度合併檔、長文獨立檔或里程碑檔。
5. 若季度檔不存在，自動建立 `<Ticker>_筆記_YYYYQ<N>.md`。
6. 依 L2 / L3 / L4 更新該股儀表板，不把 L2 小訊號硬寫成立場轉變。
7. 更新該股 index。
8. 不重複貼整篇 KOL 主整理，只萃取與該股投資論述有關的內容。

若只是 L1 提及，只更新 `indexes/ticker_index.md`，不寫入 `Stocks/<Ticker>/`。

## 7. 品質檢查

交付前檢查：

- ticker 是否粗體大寫
- 價位是否用反引號
- 推估資訊是否標「推估」
- 引文是否有來源
- source_id 是否寫入筆記與 source index
- raw 路徑 / URL 與 OCR 狀態是否記錄
- 是否已確認文章未截斷，且核心內容足以可靠整理
- 價位性質是否分清楚
- KOL 觀點與群友觀點是否分清楚
- tags 是否優先使用 `docs/TAG_TAXONOMY.md`
- 有無跨 KOL 混寫
- 有無不必要買賣建議

## 8. 回覆使用者

回覆格式：

1. 一句話說明本次入庫內容
2. 列出新增 / 更新檔案
3. 列出更新索引
4. 提醒重要風險或資料限制

若因截斷或內容不清楚而終止入庫，只回覆：

1. `內容疑似截斷 / 不清楚，已停止紀錄。`
2. 判斷原因。
3. 需要使用者補充或重傳的資料。

不要在對話中貼完整長文，除非使用者要求。
